# Implementation Plan 0002 — Catalog extensibility, manifest provenance, and the single-tool guard

This plan implements the design in
[extensibility.md](../design/extensibility.md), together with the supporting
CLI and manifest changes documented in
[README.md §5.2](../design/README.md) and
[updates.md §1](../design/updates.md). It turns `cargo-anvil` from a single
hard-wired tool into the base instance of a reusable engine: downstream teams
ship their own cargo subcommand with a customized **catalog** while reusing the
entire update engine.

Companion docs:
- [extensibility.md](../design/extensibility.md) — the catalog model, the public
  artifact API, backends, multi-level chains, the provenance/guard tie-in (§5.1).
- [updates.md](../design/updates.md) — manifest schema (`tool` / `tool_version` /
  `catalog_checksum`), the single-tool guard, the catalog checksum.
- [README.md](../design/README.md) — CLI surface (`--force`, `--version` checksum).

The design is **already merged as documentation**; this plan is the code work to
make the implementation match it. No new user-facing surface is invented here
beyond what those docs describe.

---

## Sequencing principles

1. **Refactor invisibly first, expose second.** The whole catalog model can be
   introduced internally and wired through `build_plan` with **zero change to
   emitted output** — the existing snapshot and fixture tests are the safety net.
   Only once the engine is data-driven do we expose the public `Catalog` API.
2. **Byte-identical base catalog is the invariant.** Every commit through Phase 1
   must leave `Catalog::anvil()`'s output identical to today's. A snapshot diff is
   a bug, not an expected churn.
3. **Every commit leaves the crate compiling** and the base tool behaving
   identically, so the series can land incrementally.
4. **Manifest provenance and the guard land as a coherent group** (Phase 2) — the
   field split, checksum, guard, and flags reference each other.
5. **No premature generality.** Backends stay a closed engine enum; the engine
   stays in the `cargo-anvil` crate (no `anvil-core` split) per the design
   decisions in [extensibility.md §10](../design/extensibility.md).

## Phase 0 — The internal artifact model (no behavior change)

### Commit 1: artifact types

- **New module** `catalog::artifact` (or `artifact.rs`): `Artifact` enum
  (`OwnedFile(OwnedFileSpec)`, `Region(RegionSpec)`), `OwnedFileSpec { path,
  body, gate: Option<Backend> }`, `RegionSpec { host: HostSelector, id: RegionId,
  body, syntax }`, `RegionId(&'static str)` newtype (`const fn new`),
  `HostSelector { Path(String), EachMemberManifest }`.
- **`Backend`**: reuse the existing closed enum in `backend.rs`; re-export it from
  the catalog module. No new variants.
- **Identity**: an *internal* `ArtifactKey` (`OwnedFile(path)` / `Region { host,
  id }`) used only for dedup and override; **not** `pub`. `Artifact::key()` is
  `pub(crate)`.
- **`Artifact::with_body(self, impl Into<String>)`** — returns the artifact with
  its body replaced, every other field preserved (the override primitive).
- Constructors `Artifact::owned_file`, `backend_file`, `region`, `member_region`.
- Unit tests: `with_body` preserves path/gate/host/id/syntax; `key()` identity for
  each kind; `member_region` sets `EachMemberManifest` + `Hash`.
- Design refs: [extensibility.md §4, §4.2](../design/extensibility.md).

### Commit 2: assemble the anvil catalog as data

- **New** `catalog::anvil_artifacts() -> Vec<Artifact>`: build the full base set
  from the existing embedded templates — the `justfiles/anvil/` tree
  (`mod`→`entry`, `versions`, `tools`, `checks`, `groups`, `tiers`), the managed
  regions (`justfile_imports`, `workspace_lints`, `member_lints` as
  `EachMemberManifest`, `deny`, `rustfmt`, `delta`, `spellcheck`, `clippy`), and
  the gated backend files for `github` / `ado` (composite actions / step
  templates, workflows / stages, root files, **including the per-group fan-out
  expanded to concrete paths** — replacing the runtime `render_group_action` /
  `__GROUP__` substitution with one `OwnedFileSpec` per group).
- The bodies reuse the same `include_str!` constants already in `emit::*`.
- **Test**: assert the data set's `(path, body)` pairs and region `(host, id,
  body)` triples exactly equal what the current `plan_*` emitters would render
  (a transitional equivalence test that can be deleted once Commit 3 lands).
- No wiring yet; `build_plan` still uses the old path.
- Design refs: [extensibility.md §4.1, §4.3](../design/extensibility.md).

### Commit 3: data-drive `build_plan`

- **`run::build_plan`** iterates `anvil_artifacts()` instead of calling the fixed
  list of hand-named emitters. Each artifact dispatches to the existing
  `plan_owned_file` / `plan_managed_region` drivers; `EachMemberManifest` expands
  against the discovered workspace members; `OwnedFile` artifacts whose `gate` is
  `Some(b)` are emitted only when `b` is in the resolved backend set.
- The old per-emitter entry points (`plan_local_just_tree`,
  `plan_justfile_imports`, `plan_cargo_lints`, `plan_shared_configs`,
  `plan_github_backend`, `plan_ado_backend`) collapse into the catalog data;
  delete or reduce them to thin helpers as appropriate.
- **Invariant**: snapshot + fixture tests pass unchanged (byte-identical output).
  This commit is the linchpin — its safety net is the existing test suite.
- Design refs: [extensibility.md §4](../design/extensibility.md).

## Phase 1 — The public catalog API

### Commit 4: `CliMeta`, `Catalog`, `CatalogBuilder`

- **`CliMeta { subcommand, bin_name, about, version }`** (defaults: `bin_name =
  cargo-{subcommand}`).
- **`Catalog { cli: CliMeta, artifacts: Vec<Artifact> }`** with `Catalog::anvil()`
  (CliMeta for `anvil` + `anvil_artifacts()`), `Catalog::builder(CliMeta)`,
  `Catalog::into_builder()`.
- **`CatalogBuilder`**: `subcommand` / `about` / `version`; the three uniform
  verbs `with_artifact` (add — error if identity present), `replace_artifact`
  (override — error if absent), `without_artifact` (remove — error if absent);
  `build() -> Result<Catalog, AppError>` validating those invariants.
- Public artifact constructors (`Artifact::owned_file` / `backend_file` / `region`
  / `member_region`) become `pub`.
- Unit tests for each verb's add/override/remove invariants and error cases.
- Design refs: [extensibility.md §4, §5, §10](../design/extensibility.md).

### Commit 5: the public `artifacts::` registry

- **`pub mod artifacts`** with `justfile::{entry,versions,tools,checks,groups,
  tiers}`, `region::{justfile_imports,workspace_lints,member_lints,deny,rustfmt,
  delta,spellcheck,clippy}`, `github::{…}`, `ado::{…}` — each a `fn() -> Artifact`
  returning the corresponding built-in (the same data from Commit 2, now public,
  one source of truth).
- rustdoc on each, naming the path / `(host, id)` it produces.
- Test: every registry function's `key()` is present in `Catalog::anvil()`.
- Design refs: [extensibility.md §4.1](../design/extensibility.md).

### Commit 6: thread the catalog through the engine; `run_app`

- **`run(catalog, cli)`**, **`run_update(catalog, cli, start_dir)`**, and
  **`Cli::parse_from_cargo_args(catalog, args)`** (strips `catalog.cli.subcommand`;
  renders help/about/version from `CliMeta`). `build_plan` takes the catalog.
- **`run_app(catalog) -> ExitCode`**: the body of today's `main.rs` (tracing
  init + parse + `Ok/Err → ExitCode` + dry-run exit-1), generalized.
- **`main.rs`** collapses to
  `fn main() -> ExitCode { cargo_anvil::run_app(Catalog::anvil()) }`.
- Existing CLI/integration tests updated to pass `Catalog::anvil()`.
- Design refs: [extensibility.md §5](../design/extensibility.md).

## Phase 2 — Manifest provenance, catalog checksum, and the guard

### Commit 7: split `rendered_by` into provenance fields

- **`manifest.rs`**: replace `rendered_by: Option<String>` with
  `tool: Option<String>`, `tool_version: Option<String>`, `catalog_checksum:
  Option<String>`. Serialize as three top-level keys (after `version`, before the
  `[[file]]` arrays); parse them.
- **Legacy fallback**: a lock that has only `rendered_by` still parses (split into
  tool name + version for display); absence of `tool` is treated as "no recorded
  tool" — it never triggers the guard, so upgrading an existing repo is seamless.
- Tests: round-trip; legacy `rendered_by` parse; deterministic ordering.
- Design refs: [updates.md §1](../design/updates.md).

### Commit 8: catalog checksum

- **`Catalog::checksum() -> String`** (`sha256:…`): canonical serialization over
  every artifact's identity + rendered body in sorted order (reuse
  `checksum::checksum_str`). Deterministic and independent of any repo.
- **`Plan::apply`** records `tool` = `catalog.cli.subcommand`, `tool_version` =
  `catalog.cli.version`, `catalog_checksum` = `catalog.checksum()` (threaded in
  from `run_update`, replacing the `env!("CARGO_PKG_*")` literal).
- Tests: checksum is stable across insertion order; changes when any artifact's
  path/body/gate or any region id/host changes; unaffected by `CliMeta`.
- Design refs: [updates.md §1 "Catalog checksum"](../design/updates.md),
  [extensibility.md §5.1](../design/extensibility.md).

### Commit 9: the single-tool guard + `--force`

- **`Cli`**: add `--force` (bool).
- **`run_update`**: after loading the manifest, if `manifest.tool` is `Some` and
  `!= catalog.cli.subcommand` and `!args.force`, return the refusal error (naming
  the owner and the current tool; suggesting "run '<owner>' instead, or `--force`
  to switch"). The check is content-free and fires before planning; it applies to
  `--dry-run` too (no analysis, no write).
- With `--force` (or matching/absent `tool`) the run proceeds as a **normal
  update** — orphans from the previous tool's catalog reconcile via the standard
  §5.1 rules (`Remove` if untouched, `OrphanedKept` if customized) and the
  provenance fields are rewritten to the new tool on save.
- Tests: match → proceed; mismatch → refuse (and under `--dry-run`); mismatch +
  `--force` → proceed and rewrite provenance; first run / legacy lock → proceed.
- Design refs: [updates.md §1 "The single-tool guard"](../design/updates.md),
  [README.md §5.2](../design/README.md).

### Commit 10: `--version` shows the catalog checksum

- Render `--version` (clap `long_version`, or a version string built from the
  catalog) as two lines: `cargo-anvil <version>` then `catalog: sha256:…`,
  computed from the passed catalog so forks get their own value.
- Help text for `--version` notes the second line is the catalog checksum.
- Test: `--version` output contains the catalog checksum and exits 0 without
  touching the repo.
- Design refs: [README.md §5.2](../design/README.md).

## Phase 3 — Extensibility verification

### Commit 11: second-front-end fixture and chain test

- An in-repo example catalog (`Catalog::anvil().into_builder()` with subcommand
  `demoforge`, one extra owned file, one overridden built-in via
  `artifacts::…().with_body(…)`, one dropped artifact, and one
  `member_region`) exercised by a fixture test asserting: the subcommand parses;
  the extra file is emitted; the override/drop take effect; the member region
  fans out across all members; and the **base-catalog snapshot is unchanged**.
- A `tool` field of `demoforge`, plus a separate test that running `anvil` against
  a `demoforge`-owned lock refuses, and `--force` switches it.
- A minimal multi-level test (`demoforge` re-extended) confirming
  `into_builder()` composes and override/remove address ancestor artifacts.
- Design refs: [extensibility.md §7, "Verification"](../design/extensibility.md).

### Commit 12: crate docs / README

- Update the `lib.rs` crate-level docs to cover `Catalog`, `run_app`, the
  `artifacts::` registry, `--force`, and the `--version` checksum.
- Regenerate the crate README via `just readme` (generated; do not hand-edit).
- Run `just format` and `just spellcheck`.

## Out of scope for this plan

- **Fork-authored backends.** The backend set stays the closed `{github, ado}`
  enum; forks override / drop / add *files gated on an existing backend*, never a
  new backend ([extensibility.md §9](../design/extensibility.md)).
- **Per-fork on-disk rebranding.** The `anvil` namespace (`.anvil.lock`,
  `anvil-managed`, `justfiles/anvil/`, `anvil-` recipes) is the fixed engine
  format; nothing here parameterizes it.
- **`anvil-core` crate split.** Engine + API stay in `cargo-anvil`
  ([extensibility.md §10](../design/extensibility.md)).
- **Runtime plugins / dynamic catalog loading.** Catalogs are compiled-in Rust.
- **Tier-2 check-level catalog editing.** Forks replace whole `justfile` artifacts
  (e.g. `checks.just`) wholesale; structured check generation is not in scope
  ([extensibility.md §6](../design/extensibility.md)).

## Risk and rework hot-spots

- **Commit 3 (`build_plan` rewrite) is the linchpin.** It must produce
  byte-identical output; the entire snapshot + fixture suite is its safety net.
  Land it only when those tests are green, and keep the Commit 2 equivalence test
  until then.
- **Per-group backend fan-out.** Moving from runtime `__GROUP__` substitution to
  one concrete `OwnedFileSpec` per group is the most error-prone part of Commit 2
  — the group list and emitted paths must match `github.rs` / `ado.rs` `GROUPS`
  exactly, or snapshots break.
- **Legacy locks vs. the guard.** A pre-split lock has no `tool` field; treating
  its absence as "no owner" (Commit 7/9) is what keeps the guard from spuriously
  refusing on the first post-upgrade run. Do not derive a `tool` from the legacy
  `rendered_by` string for guard purposes.
- **Checksum canonicalization (Commit 8).** The serialization feeding the hash
  must be order-independent and stable across releases, or every run would report
  a spurious catalog change. Pin it with a determinism test, and treat any change
  to the canonical form as a breaking lock-churn event.
- **`Plan::apply` provenance threading.** Apply currently hard-codes
  `env!("CARGO_PKG_*")`; Commits 8–9 must route tool identity from the catalog
  instead, or a fork would stamp `cargo-anvil` into its own locks.
